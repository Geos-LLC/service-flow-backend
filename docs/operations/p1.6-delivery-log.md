# P1.6 — Unified Delivery Log

**Status:** Live on staging (commit at P1.6 merge).
**Constitution refs:** §0 P2 (loud failures), §9 P1.6 (webhook_delivery_log requirement).
**Companion:** [synchronization-constitution.md](../architecture/synchronization-constitution.md), [p0-monitoring.md](./p0-monitoring.md).

## Why this exists

Before P1.6, answering "did this message get through?" required checking 6+ different surfaces:

| Source | What it stored | Tenant scope | Operator queryability |
|---|---|---|---|
| `notification_email_logs` | every email send | per-tenant | tenant-scoped endpoint only |
| `leadbridge_outbound_events` | SF→LB queue + delivery state | per-tenant | no operator UI; SQL only |
| `zb_sync_dirty` | downstream ZB sync failures | per-tenant | `GET /api/zenbooker/sync-dirty` |
| `ledger_drift_detected` | settled-ledger divergence | per-tenant | SQL only |
| `payment_reconcile_runs` / `_catches` | hourly ZB reconcile audit | per-tenant | `GET /api/zenbooker/payment-reconcile-log` |
| `communication_webhook_events` | LB inbound dedup | per-tenant | no operator UI |
| Loki streams | `[ZB-auth-observe]`, `[ZB-atomic]`, `[ZB-dirty]`, `[Boundary]`, `[WhatsApp-status]`, `[NotificationEmail]` | text logs | Grafana, no DB join |

`delivery_log` is the **unified queryable surface**. Every notable cross-system delivery — outbound or inbound — writes one row, and operators query one place for cross-system audit.

## Schema (migration 042)

```sql
CREATE TABLE delivery_log (
  id                   BIGSERIAL PRIMARY KEY,
  user_id              BIGINT NULL,           -- tenant scope (NULL = platform/system)
  source_system        TEXT NOT NULL,         -- 'service_flow' | 'leadbridge' | 'sigcore' | 'zenbooker' | 'sendgrid' | 'stripe' | 'whatsapp' | 'twilio' | 'openphone'
  destination_system   TEXT NOT NULL,         -- same vocabulary
  channel              TEXT NULL,             -- 'email' | 'webhook' | 'sms' | 'whatsapp' | 'voice' | 'api_rpc'
  event_type           TEXT NOT NULL,         -- 'email.invoice', 'lb_outbound.lead.status_changed', 'zb_inbound.job.completed', etc.
  correlation_id       TEXT NULL,             -- cross-system tracking id
  request_id           TEXT NULL,
  payload_hash         TEXT NULL,             -- SHA-256 hex
  delivery_direction   TEXT NOT NULL,         -- 'outbound' (we sent) | 'inbound' (we received)
  status               TEXT NOT NULL,         -- 'queued' | 'sent' | 'delivered' | 'failed' | 'rejected' | 'rate_limited' | 'duplicate' | 'timeout'
  response_code        INTEGER NULL,
  latency_ms           INTEGER NULL,
  retry_count          INTEGER NOT NULL DEFAULT 0,
  provider             TEXT NULL,
  provider_message_id  TEXT NULL,
  error_message        TEXT NULL,
  error_class          TEXT NULL,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  resolved_at          TIMESTAMPTZ NULL,
  context              JSONB NULL
);
```

Indexes target operator query patterns: tenant (`user_id, created_at`), correlation drilldown (partial), failure filter (partial), edge (`source, destination`), direction+channel.

## Coexistence strategy

`delivery_log` does **not** replace the existing per-domain tables. Each remains canonical for its own per-domain queries:

| Domain table | Stays canonical for | `delivery_log` provides |
|---|---|---|
| `notification_email_logs` | per-tenant email viewer; SendGrid-specific fields (recipient_name, subject, metadata) | unified cross-system view; correlation across email + other channels |
| `leadbridge_outbound_events` | retry queue, lease/claim state, defer_reason, payload_json | terminal-state audit + cross-tenant operator view |
| `zb_sync_dirty` | per-job dirty marker (idempotent unique row per `(user, target, operation)`) | the unified surface where ZB dirty failures appear alongside email failures and LB failures |
| `ledger_drift_detected` | financial-integrity signal (immutable) | not duplicated — it's a different kind of audit (settled state, not delivery) |
| `payment_reconcile_runs`/`_catches` | per-run sweep state + caught payments | not duplicated — reconcile is internal projection, not cross-system delivery |
| `communication_webhook_events` | LB inbound dedup keyed on `event_id` | LB→SF inbound delivery audit (planned, not in P1.6 first round) |

**Decision rule:** if it's a *delivery* (data leaving or entering this service via an external boundary), `delivery_log` gets a row. If it's an *internal projection* (rebuild, drift detection, reconcile), it stays in its canonical table only.

## Writers wired in P1.6

| Site | What writes | Direction | event_type prefix |
|---|---|---|---|
| `notification-email.service.js` `logSend()` | every email send (sent/failed/rate_limited) | outbound | `email.` |
| `workers/leadbridge-outbound-drainer.js` `logLine()` | every terminal LB outbound transition (sent/dlq/skipped) | outbound | `lb_outbound.` |
| `zenbooker-sync.js` `/webhook` POST handler | every ZB inbound webhook (per-user, per-event); plus auth-rejected events | inbound | `zb_inbound.` |

Deferred to later rounds (next P1 increment):
- Sigcore inbound webhooks (`sigcore.openphone.message.inbound`, `sigcore.whatsapp.*`)
- LB inbound webhooks (`lb_inbound.lead_status_changed`, etc.)
- SF→ZB outbound commands (when the SF↔ZB design from `docs/architecture/zb-outbound-command-confirmation.md` is implemented)
- Stripe webhook deliveries (future)

## Long-term replacement strategy

Phase chart:

| Phase | When | What changes |
|---|---|---|
| **A — Land** (P1.6 now) | this PR | `delivery_log` table + helper + 3 dual-writers + 2 operator endpoints |
| **B — Read consolidation** | next P1 round | Operator UIs migrate to read `delivery_log` instead of per-domain tables. Per-domain tables remain canonical writers. |
| **C — Reader deprecation** | when no consumer reads from a per-domain table for its delivery view | Per-domain table loses its tenant-scoped read endpoint (e.g. `GET /api/notification-email/logs` redirects to `/api/delivery-log?channel=email`) |
| **D — Writer evaluation** | when a domain table's only consumer is the audit dual-write itself | Replace the table with a generated view over `delivery_log`. E.g. `notification_email_logs` → `CREATE VIEW notification_email_logs AS SELECT ... FROM delivery_log WHERE channel='email'`. |
| **E — Source consolidation** | when all per-domain writers route through `logDelivery` | Domain-specific fields move into `context jsonb`. Domain tables are dropped. |

Phases C/D/E are not part of P1.6. They are documented here so the migration path is clear and incremental — no big-bang rewrite.

**Never replace `ledger_drift_detected`.** That table records financial-integrity drift and is a Constitution §3.1 monitoring surface. It is NOT a delivery audit — it represents a different invariant. Keep it as-is forever.

## Loki anchor

Every `logDelivery` call emits one `[DeliveryLog]` line BEFORE the DB write. The line is the floor of observability — even if the DB insert fails, this stays in Loki:

```
[DeliveryLog] user_id=2 source=service_flow dest=sendgrid dir=outbound channel=email event=email.invoice status=sent code=202 corr=abc123 latency_ms=180
[DeliveryLog] user_id=2 source=zenbooker dest=service_flow dir=inbound channel=webhook event=zb_inbound.job.completed status=sent latency_ms=42
[DeliveryLog] user_id=null source=zenbooker dest=service_flow dir=inbound channel=webhook event=zb_inbound.auth_rejected status=rejected code=401 error_class=missing_signature
```

Failed/rejected/timeout statuses emit at `warn` level; successful at `log` level. Alert rules in `p0-monitoring.md` should be extended to cover `[DeliveryLog]` matches (TODO P2).

## Operator endpoints

| Endpoint | Auth | Scope | Use case |
|---|---|---|---|
| `GET /api/admin/delivery-log` | `authenticateAdmin` | cross-tenant | platform operator audit, escalation triage |
| `GET /api/delivery-log` | `authenticateToken` | tenant-scoped (forced) | per-tenant self-service via UI |

Common filters supported by both: `source_system`, `destination_system`, `channel`, `direction`, `status` (comma-separated for IN), `event_type_prefix`, `correlation_id`, `since`, `until`, `limit`, `offset`. Admin variant adds `user_id` as a tenant filter.

Example queries:

```
# all email sends to this tenant in the last 24h
GET /api/delivery-log?channel=email&since=2026-05-14T00:00:00Z

# every cross-system failure this hour (admin)
GET /api/admin/delivery-log?status=failed,rejected,timeout&since=...

# trace a specific message end-to-end
GET /api/admin/delivery-log?correlation_id=W12345

# all LB outbound failures awaiting operator action
GET /api/admin/delivery-log?source_system=service_flow&destination_system=leadbridge&status=failed
```

## Tenant isolation

The tenant-scoped endpoint `/api/delivery-log` forces `user_id = req.user.userId` AFTER applying caller filters, so a `?user_id=999` query param cannot escape the scope. Admin variant requires `authenticateAdmin` (the existing `ADMIN_EMAIL` / `ADMIN_PASSWORD` env-var-gated admin JWT).

The helper itself does not enforce scope — it accepts whatever `userId` the caller passes. Callers are responsible for passing the correct tenant id, just as they are for `markDirty`, `recordLedgerDrift`, and the existing audit writers. The constitution requires this (§0 P3) and the existing source-text scan tests enforce it for the established surfaces.

## Future extension: SF↔ZB command confirmation

When the SF→ZB outbound command queue ships ([zb-outbound-command-confirmation.md](../architecture/zb-outbound-command-confirmation.md)), it WILL use `logDelivery` for every command lifecycle event:

```
event=zb_outbound.job.reschedule  status=queued     direction=outbound
event=zb_outbound.job.reschedule  status=sent       direction=outbound  (ZB returned 200)
event=zb_outbound.job.reschedule  status=delivered  direction=outbound  (echo webhook confirmed)
event=zb_outbound.job.reschedule  status=failed     direction=outbound  (4xx terminal)
event=zb_outbound.job.reschedule  status=timeout    direction=outbound  (confirm_timeout)
```

This means the operator dashboard answers "did the SF→ZB reschedule actually land?" from the same query surface that answers "did the invoice email send?" — one table, one shape, one Loki anchor.

## Rollback

`delivery_log` is strictly additive. Code rollback removes the writers; the table can stay (idle, queryable for historical rows). Per-domain canonical writers are unaffected by P1.6 rollback because they were never replaced — only dual-written alongside.

## What stays the same

- All existing canonical audit tables (`notification_email_logs`, `leadbridge_outbound_events`, `zb_sync_dirty`, `ledger_drift_detected`, etc.) — unchanged shape, unchanged readers, unchanged writers.
- Constitution §3.1 immutability — `delivery_log` writers cannot mutate `cleaner_ledger`; the helper inserts INTO `delivery_log` only.
- ZB auth flag (`ZB_WEBHOOK_AUTH_REQUIRED`) — unchanged.
- All SF↔ZB outbound design — still design-only.
- Email templates — preserved byte-for-byte (P1.5 work unchanged).
